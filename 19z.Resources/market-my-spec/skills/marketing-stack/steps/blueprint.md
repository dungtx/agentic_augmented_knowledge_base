# Blueprint (`/marketing-stack blueprint`)

Regenerate `marketing/infrastructure.md` from the observed state of installed recipes + the strategy.

## Phase 1 — Gather

Same as `steps/inventory.md` Phase 1. Read all detection inputs + the prior `marketing/infrastructure.md` (to preserve user-edited notes).

## Phase 2 — Compute state and fit per recipe

Same matchers as inventory. Each recipe gets:
- `state`: `ready` | `partial` | `broken` | `absent`
- `fit`: `required` | `nice-to-have` | `out-of-scope`

Preserve notes and last-verified dates from the prior blueprint when state hasn't changed.

## Phase 3 — Optionally batch-re-validate `ready` recipes

Offer:

> "Blueprint regenerates from observed state. Want to re-verify all `ready` recipes by running their validation specs? (~1 minute total; y/n)"

If yes, run the `validation:` block from each `ready` recipe's frontmatter. Update state to `broken` if a validation fails.

If no, trust prior state.

## Phase 5 — Close

> "Blueprint written: `marketing/infrastructure.md` — N ready, M partial, K absent (required). `/daily-plan` reads this when picking activities."

## Anti-patterns

- **Overwriting user-edited notes.** Notes column is preserved across regenerations. Never clobber.
- **Generating when nothing changed.** If observed state exactly matches the prior blueprint, tell the user "no changes detected; blueprint untouched."
- **Hiding broken recipes.** Surface explicitly in Open issues + Next actions.
- **Example narrative in output.** The template uses `<date>` placeholders and generic strings. When writing real state, use actual values — never leak example rows into the real file.
- **Dropping columns.** State and Fit are orthogonal and both required. Never collapse to a single "status."
